# Дизайн монитора лаборатории (Доминика)

_Статус: v1.0 (черновик зафиксирован 2026-07-12). Автор: Доминика. Источник решений: обсуждение с ЗавЛабом + /research по best-practice (SRE/status-report)._

## 1. Назначение

Доминика = монитор лаборатории. Ежечасно приносит сводку о состоянии лаборатории и при проблемах **советует** (не действует без «го») запустить расследование релевантным агентом.

## 2. Принцип ливенесса

- Ежечасный ответ монитора САМ ПО СЕБЕ = доказательство, что OpenClaw жив.
- Нет ответа в час X → 99%, что платформа мертва (gateway / планировщик / агент упали).
- Следовательно: сам факт прихода отчёта = верхнеуровневый health-check всей системы.

## 3. Категории мониторинга (8)

1. **Агенты колонии (8):** жив/активность/сессия/ошибки; grimoire.md жив; его кроны (вкл/выкл, статус, consecutive errors)
2. **Платформа OpenClaw:** gateway active / NRestarts / версия; телеграм-боты (8) шлют ли, не забанены ли; раз в сутки `openclaw doctor` + бэкап при valid
3. **MCP-сервисы:** mcp-heartbeat(:8088) / mcp-memory(:8087) / mcp-apikeys(:8086) active + порт; выключенные (filesystem/lab-research/shell/status) — флаг, если вдруг active без спроса
4. **Память и поиск:** ONNX :8082 /health; lab_search (FAISS) векторов + health end-to-end; reindex-таймеры active
5. **Данные и хранилища:** PostgreSQL(api-hub-db) up/размер/бэкап; SQLite state не заблокирована; диск корень <85%, /tmp, логи, бэкапы
6. **Сеть и внешний доступ:** VPN(amnezia-awg2) up; searxng отвечает; внешние API(tavily/firecrawl/…) live_status; SSL-сертификаты до expiry < N дней
7. **Проекты и код:** git-статус ключевых проектов; открытые INC-*; CVE/dependency
8. **Сервер (бывш. Ресурсы хоста):** CPU / RAM / диск; Docker-контейнеры running vs простаивают (кандидаты на удаление, красная линия #26)

## 4. Формат отчёта

- **Шапка — гибрид:** `🦊 ЛабМонитор · ЧЧ:ММ МСК · ТРЕВОГА/OK` всегда; отдельный блок «🔴 провалы: …» — ТОЛЬКО при наличии 🔴 (при зелёном дублей нет). Источник: PMI/tempo/fanruan — консистентный формат + проблемы сразу на виду.
- **Ежечасная сводка:** 8 строк OK/FAIL по категориям + лёгкие проверки (gateway active, бот шлёт, порт отвечает).
- **Тяжёлые проверки** (доктор конфига + бэкап + git-дамп): **раз в сутки**. Источник: Uptime.com/MTTD — тяжёлое реже, чтобы не плодить шум/нагрузку.
- **Варнинги доктора — дрейф:** храним baseline варнингов, 🔴 только при РОСТЕ/НОВОМ варнинге; базовые шумные (message-tool-unavailable, legacy config-health) — в allowlist, не светим каждый час. Источник: DevOps-2025/SRE — deduplication, dynamic baselines, фильтр известного шума (alert fatigue).
- **Все ✅:** слать ПОЛНУЮ 8-строк сводку каждый час (никакого «минимума»). Источник: PMI/knack — консистентность; плюс принцип ливенесса.
- **Триггер полного дампа:** «подробно» (командная форма «!подробно!»). Матчинг: сообщение содержит токен «подробно» (без учёта регистра); предпочтительно как команда с префиксом «!», чтобы не срабатывать на слово «подробно» в обычной переписке. Полный дамп = логи, per-agent детали, все варнинги доктора, диск по разделам.
- **Без дубля данных в `--full` (2026-07-13):** полный дамп не повторяет одни и те же данные в нескольких местах.
  - Заголовок OK-категории = только `✅ N. Имя` (числа живут в деталях `·` ниже).
  - Доктор-предупреждения (`openclaw doctor`) и самопроверка движка — ЕДИНСТВЕННЫЙ источник: выделенная секция 🩺 внизу полного дампа. В [2] OpenClaw они НЕ дублируются (ни в деталях, ни во вторичной ⚠️-строке). В коротком отчёте самопроверка остаётся в summary [2] (там нет низа).
  - Диск % — ЕДИНСТВЕННЫЙ источник: выделенная секция 💾 Диск внизу. В [5] Данные строка `disk /:` убрана (summary [5] в коротком отчёте диск оставляет для health-сигнала + self_factcheck).
  - Упавшая категория держит summary в заголовке (`🔴 N. Имя — <причина>`).

## 5. Слой активного реагирования

- **Автономия = СОВЕТУЕТ (advise).** При новом 🔴 монитор НЕ спавнит агента без «го». Он предлагает: «рекомендую спавн <агент> с набором <скиллы> для расследования <проблема>» и ждёт подтверждения. Любые правки продакшена — только после «го».
- **Триггер слоя:** появление нового 🔴 (drift, не базовый шум).
- **Маршрутизация проблема → агент:**
  - Поиск/память (lab_search, FAISS, ONNX) → Муравей / Ворон
  - Gateway/платформа → Котолизатор / Муравей
  - Безопасность/SSL/сертификаты → Сова
  - Внешние API/ключи → Бестия / Ворон
  - Упал конкретный агент → тот агент + Мангуст
  - Проекты/git/код → Штрейкбрехер
- **Рецепт расследования (набор скиллов спавнящегося агента):**
  - «Глубокое погружение в проблему» → навык `research` (глубокое веб-погружение + fact-check)
  - «Археолог корней проблем» → root-cause методология: `labsearch` (похожие инциденты в доках лаборатории) + `research` (корни в интернете) + чтение логов/trajectory
  - `labsearch` → семантический поиск по документации лаборатории
  - `research` → веб + fact-check
  - **Цель:** найти паттерны и инсайты — в документации лаборатории И в интернете.
- **Guardrails (обязательно):**
  - Дедуп по ключу инцидента + cooldown (не спавнить тот же 🔴 чаще раза в N часов) — защита от loop
  - Circuit breaker / anti-loop: стоп после K безуспешных итераций
  - Расследование **read-only** (только чтение + research), без правок продакшена
- **Куда класть инсайты расследования:** TBD (MEMORY монитора / файл инцидента / отдельный `lab-insights`).

## 9. Гард честности (self-factcheck) — внедрено 2026-07-13

Корень вранья (archaeologist): монитор был источником правды сам для себя, фактчек
был внешним и разовым. Решение — встроить фактчек в ядро.

- `THRESHOLDS` — все пороги (disk%, load×ядра, NRestarts) вынесены в блок констант
  с указанием источника нормы. Ноль магических чисел «знаю, сколько должно быть».
- `self_factcheck(results)` — всегда включён в `build_report`. Сверяет заголовок
  (summary/✅) с деталями и порогами. Любое противоречие → строка
  `🔴 САМОПРОВЕРКА (монитор поймал сам себя)` в ежечасном отчёте.
- `independent_probe()` + режим `--selftest` — независимый замер каждой категории
  другим кодом; монитор печатает «монитор говорит» vs «независимо» и вердикт.
- Живые источники только: systemd, df, docker, lab_search health, openssl, /proc.
  Никаких хардкодов ожидаемого количества (MCP/агенты считаются из системы).

Проверено: при искусственной лжи (6 случаев: отсутствующий файл, NRestarts=9,
MCP 2/3, disk 96%, load 9.5, DOWN в деталях) гард поймал все 6.

## 10. Принцип живости (правило ЗавЛаба, 2026-07-13)

- Пришёл ежечасный отчёт ⇒ OpenClaw и агент-отправитель живы 100%.
- Не пришёл отчёт ⇒ мертвы 100% (конвейер gateway→агент→Telegram не сработал).
- Следствие: отдельный пинг живости агентов НЕ нужен — доставка сама доказывает.
- Категория 1 (Агенты) поэтому = всегда ✅ при доставке; проверяет лишь целостность
  файлов-памяти (grimoire), НЕ живость. self_factcheck не путает отсутствие файла
  с «мёртв».
- Для «не пришёл = мертвы» требуется ВНЕШНИЙ watchdog (проверка: был ли отчёт за
  последний час) — вне этого скрипта, опционально.

## 11. Watchdog — НЕ ДЕЛАТЬ (решение ЗавЛаба, 2026-07-13)

Предложенный внешний watchdog (сторож на отсутствие отчёта) ОТВЕРГНУТ как избыточный.
Обоснование ЗавЛаба:
- Он САМ — внешний сторож (телефон вне сервера). Пишет агенту → ответил = OpenClaw жив.
  Не ответил = сломалось. Ручная проверка = самый надёжный liveness-детектор.
- «Агенты отвечают = OpenClaw жив» — финальная формулировка правила живости
  (любой ответ агента в Telegram доказывает, что конвейер gateway→агент→Telegram работает).
- Автоматический watchdog не нужен: пришёл отчёт = жив (доставка доказана),
  не пришёл = ЗавЛаб проверит рукой за 5 секунд.

## 13. Логика перезапусков gateway (переделано 2026-07-13, команда ЗавЛаба)

**Проблема старой логики.** Категория [2] OpenClaw мерила накопленный `NRestarts`
(systemd-счётчик за всю жизнь юнита) и сравнивала с хардкод-порогом `nrestarts_ok=5`.
Это давало две проблемы:
- Тревога 🔴 **горела вечно** после достижения порога, даже если gateway стабилен неделю.
- Путаница «loop сейчас» vs «редко падал в истории»: накопленный счётчик не
  различает их. При этом активный restart-loop systemd ловит САМ (юнит ушёл бы в
  `failed` по `StartLimitBurst=5 / StartLimitIntervalUSec=10s`), так что живой gateway
  с `NRestarts=6` — это НЕ loop, а редкие падения, разнесённые во времени.

**Новая логика (оконная).** Отчёт монитора приходит **каждый час** (cron
`heartbeat-dominika`, `0 * * * *` MSK). Значит окно отчёта = **1 час**, и за это окно
ЗавЛаб может сверить перезапуски с памятью: «я сам рестартил в этом часу? да → ок.
нет → подозрительно, расследуем».

- Источник данных: `journalctl -u openclaw-gateway.service --since '-1h'`.
- Классификация (чистая функция `classify_restarts`):
  - `total` = число стартов (`Started|Starting OpenClaw Gateway`) за окно.
  - `auto` = число `Scheduled restart` (маркер авто-восстановления systemd
    `Restart=always` после падения).
  - `manual` = `total − auto` (ручные рестарты ЗавЛаба / загрузка сервера).
- Пороги (`THRESHOLDS`):
  - `restart_window = "1h"` — окно совпадает с окном отчёта.
  - `nrestarts_window_auto_ok = 0` — авто-перезапусков за окно: норма 0;
    `>=1` → 🔴 (gateway сам падал, нежданно).
  - `nrestarts_ok = 5` — **оставлен** для ПРОЧИХ сервисов (накопленный
    lifetime-порог); для gateway больше не управляет тревогой.
- Вывод в отчёте:
  - 0 за час → ✅ `перезапусков за 1h: 0`.
  - ручные (>0, auto=0) → 💡 `ручных рестартов за 1h: N (сверься: ты сам рестартил?)`.
  - авто (>=1) → 🔴 `АВТО-перезапусков за 1h: N (systemd сам поднимал — подозрительно)`.
  - `gateway DOWN` → 🔴 (это и есть случай, когда systemd не смог поднять / ручной stop).
- Исторический `NRestarts` (lifetime) показывается в блоке деталей для справки,
  но **НЕ управляет тревогой**.

**Обоснование нормы <5 (старой) признано.** Порог `nrestarts_ok=5` был эвристикой
разработчика без внешнего источника (в отличие от `disk_warn_pct` из ADR-039).
В оконной логике норма обоснована: окно = час (окно отчёта), авто-перезапуск =
падение, которого в норме быть не должно; ручные рестарты ЗавЛаб ожидает/знает.

**Формат эмодзи (требование ЗавЛаба 2026-07-13):** все эмодзи в отчёте — в начале
строки. Поэтому статус-эмодзи (✅/🔴) ставится ТОЛЬКО в слое рендера (`build_report`:
`{name}: ✅ ...` в коротком, `✅ 2. OpenClaw — ...` в полном), а `detail` категории НЕ
содержит эмодзи посреди текста. Самопроверка движка (`⚠️ самопроверка: ...`) выносится
на ОТДЕЛЬНУЮ строку, начинающуюся с ⚠️ (через `\n` внутри detail). Это убирает эмодзи
внутри описательной строки (было: `... 6 (норма <5) ⚠️ самопроверка: ...`).

**Покрытие тестами:** `test_classify_restarts_ok / _manual / _auto`,
`test_thresholds_extended` (проверяет `restart_window`, `nrestarts_window_auto_ok`,
сохранение `nrestarts_ok`).

## 14. Вариант A: понятные summary + details (2026-07-13)

По команде ЗавЛаба категории [7] Проекты и [8] Сервер сделаны человекочитаемыми
(коммиты 2960dcd, 51fd4ef).

**Категория [7] Проекты:**
- Короткий отчёт: `✅ Проекты: незакоммичено N файлов (repo список) — не сбой; инциденты X открыто / Y (Z закрыто, P% решено)`
- `!подробно`: разбивка по репозиториям (включая 0/чистый) + топ-5 старейших открытых инцидентов (владелец, возраст в днях).
- «git-dirty» → «незакоммичено», «рабочая норма» → «не сбой».

**Категория [8] Сервер (бывш. Хост):**
- Короткий отчёт: `✅ Сервер: нагрузка CPU X из N ядер (~P%, статус); память A/B MB (доступно C MB); контейнеры K запущено`
- `!подробно`: loadavg по минутам, RAM (used/free/buff/cache/available)+свободно, список контейнеров по именам, ядра CPU с %.
- `total = used + free + buff/cache` (buff/cache — reclaimable; available ≈ free + reclaimable). Монитор показывает все 4 поля, чтобы уравнение сходилось на глаз.
- Переименована Хост → Сервер (CATEGORIES + тесты).
